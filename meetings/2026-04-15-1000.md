# Rubber Duck Working Session

Date: 2026-04-15 10:00 AM

## Summary

The discussion focused on shaping the Rubber Duck workflow around
meeting minutes, backlog curation, and implementation. A key theme
was moving review and control up from code into design documents,
workflow definitions, and validator scripts so agents can operate
more reliably and at lower cost.

## Main Points

- The project direction is to connect meeting minutes, discussion
  artifacts, backlog grooming, and implementation workflows.
- Separate roles are emerging for governing docs, backlog curation,
  and implementation.
- CRISPY was discussed as a useful acronym for agent role design:
  context, role, input, steps, parameters, yield.
- Workflow structure was framed as essential for both reliability
  and reducing expensive agent reasoning.
- Validation scripts were distinguished from review. The goal is to
  prove constraints deterministically rather than rely only on LLM
  judgment.
- Human review should focus primarily on design and interface
  changes, not line-by-line code inspection.
- A possible new contributor from CS 301 was discussed for more
  general improvements and workflow refinement, while Ty continues
  focusing on technical and expansion-critical work such as Teams.
- There is outside interest in adapting Rubber Duck for the
  business school, and physics follow-up was also mentioned.

## Decisions

- Keep backlog and changelog as separate concepts: backlog for
  future work, changelog for completed changes.
- Use meeting outputs as inputs for backlog creation and curation.
- Continue emphasizing validators, tests, and workflow harnesses as
  the main way to enforce system constraints.
- Keep Ty focused on more technical platform work, especially
  Teams-related expansion needs.
