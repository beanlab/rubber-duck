# Spring Cleaning 04-23

## Test Evaluation
- Finding: The current automated suite does not cover the documented Discord-facing interface in `docs/application_interface.md`; there are no tests for startup readiness, admin command routing, message filtering, thread creation, join-link messaging, reactions, or conversation closing/error messages. Confidence: high. Suggested next step: add black-box tests around `python -m src.main` startup and Discord event handling for the documented routing/workflow contract. Complexity: high.
- Finding: The documented workflow contracts for `agent_led_conversation`, `user_led_conversation`, `conversation_review`, `registration`, and `assignment_feedback` are effectively untested. The suite only exercises helper behavior in `tests/test_dataset_tools.py`, `tests/test_python_tools_formatting.py`, and `tests/test_sql_metric_handlers.py`. Confidence: high. Suggested next step: add focused workflow tests that assert the user-visible sequences and guardrail outcomes described in the interface doc. Complexity: high.
- Finding: `tests/test_sql_metric_handlers.py` provides only indirect coverage for the `!messages`, `!usage`, and `!feedback` admin surface. It validates table persistence shape, but not command parsing, zip export behavior, operator-visible responses, or `!metrics` aggregation. Confidence: high. Suggested next step: add admin-command tests at the command-routing layer that assert the documented outputs. Complexity: medium.
- Finding: `tests/test_dataset_tools.py` and `tests/test_python_tools_formatting.py` provide only indirect coverage for the stats-oriented duck experience. They validate dataset listing and numeric rendering details, but not the documented thread workflow, attachments, prompt/response flow, or exported artifact behavior. Confidence: medium. Suggested next step: add user-level tests for the configured stats duck channel behavior and rendered outputs sent through the Discord interface. Complexity: medium.
- Finding: Several existing assertions are more specific than the documented external contract. Examples include SQL column ordering/header shape in `tests/test_sql_metric_handlers.py` and exact helper string formatting in `tests/test_python_tools_formatting.py`; these are useful internal regressions, but they do not map cleanly to promises in `docs/application_interface.md`. Confidence: medium. Suggested next step: decide whether these details belong in the public interface docs or should remain clearly labeled as implementation-level tests. Complexity: low.
- Finding: The startup and configuration contract is not covered. There are no tests for `--config`, `CONFIG_FILE_S3_PATH` fallback, recursive `$include` resolution, deep-merge behavior, or the documented `Duck online` readiness message despite those being explicit operator-facing promises. Confidence: high. Suggested next step: add config-loader and startup integration tests aligned to the documented operations interface. Complexity: high.
