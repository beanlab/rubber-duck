## Prompt Evaluation
- Finding: `socratic-duck` prompt references an "Explainer Duck" handoff, but no explainer workflow/duck is configured. Evidence: `prompts/production-prompts/socratic-duck.txt` lines referencing "Explainer Duck"; `rg -n "Explainer|explainer" production-config.yaml docs src` returned no matches. Confidence: high. Suggested: either add an explainer duck/workflow and routing or remove the handoff instructions to avoid dead-end behavior.
- Finding: `standard-rubber-duck` hardcodes April 1st behavior, which will be wrong on all other dates. Evidence: `prompts/production-prompts/standard-rubber-duck.md` line `today is April 1st`. Confidence: high. Suggested: gate the gag on the real date or move this into a seasonal/temporary prompt to avoid off-season mismatch.
- Finding: Registration prompt assumes "resume-only" flow, which conflicts with the baseline registration sequence described in the application interface. Evidence: `prompts/production-prompts/registration.txt` "Resume Assumption" and "Entry Logic"; `docs/application_interface.md` registration workflow lists initial steps (Net ID → email → nickname → roles). Confidence: medium. Suggested: clarify in config/workflow that this prompt is only used for resume cases, or add a fresh-start path to the prompt.
- Finding: `project-determiner` contains contradictory instructions for no-match cases and output format, which can misalign with assignment_feedback workflow selection. Evidence: `prompts/assignment-feedback/project-determiner.md` says "If none match, return the project name of the uploaded report" and later "return 'No matches'" while also requiring output to match provided options; examples conflict. Confidence: high. Suggested: pick one policy (e.g., strictly choose from options or return "No matches") and update text/examples accordingly.
- Finding: Stats prompts duplicate large chunks of tool/formatting guidance, increasing drift risk. Evidence: similar "Python Tool Guidelines", "Table Rendering Rules", and "Text Output" sections in `prompts/production-prompts/stats.md` and `prompts/production-prompts/cs-stats.md`. Confidence: low. Suggested: consider factoring shared guidance into a shared include or documented template to keep the two aligned.
- Finding: `agent_led_conversation` is documented as one-shot, but `standard-rubber-duck` instructs multi-turn persistence, which may conflict with the workflow contract. Evidence: `docs/application_interface.md` says "`agent_led_conversation` runs a one-shot agent response flow"; `prompts/production-prompts/standard-rubber-duck.md` requires continuing conversation and not concluding unless explicitly told. Confidence: medium. Suggested: confirm actual runtime behavior for `agent_led_conversation` and align either the prompt or the interface contract.
- Finding: Code complexity tool prompt output requirements do not match the tool contract in config. Evidence: `production-config.yaml` describes `code_complexity_agent` returning a string Big-O analysis; `prompts/312-project-design/generate-code-complexity.md` requires returning the full code with line-by-line complexity comments inside a code fence. Confidence: high. Suggested: align the tool doc_string/output expectations with the prompt (or vice versa) to avoid consumers expecting a different payload shape.
- Finding: `tier-grader-structured-output-single` examples are not valid JSON despite strict JSON schema output requirements. Evidence: `prompts/assignment-feedback/tier-grader-structured-output-single.md` examples use single quotes and Python dict style; `production-config.yaml` sets `output_format` with `strict: true` JSON schema for this agent. Confidence: high. Suggested: update examples to valid JSON (double quotes, proper braces) and reiterate strict JSON in the prompt.

## Src Evaluation
- Finding: Duplicate queue-wait logic exists in `TalkTool.receive_message_from_user` and `utils.message_utils.wait_for_message`, increasing drift risk for timeouts/queue semantics. Evidence: `src/armory/talk_tool.py` defines `receive_message_from_user` with `quest.queue('messages', None)` + `ctx.timeout`; `src/utils/message_utils.py` defines `wait_for_message` with similar queue/timeout; used in `src/workflows/registration.py` and `src/workflows/assignment_feedback_workflow.py`. Confidence: high. Suggested: consolidate into a shared helper (or document the intentional divergence) so timeouts and queue behavior stay aligned.
- Finding: Dead/unused `_tools` registry in `armory.tools` appears unreferenced. Evidence: `src/armory/tools.py` defines `_tools: dict[str, Callable] = {}`; `rg -n "_tools" -S src/armory` shows no reads/writes besides the definition. Confidence: high. Suggested: remove the unused registry or wire it into `Armory` if it is meant to cache tool metadata.
- Finding: `generate_error_message` builds a full error payload that is never sent, leaving unused data flow and confusing intent. Evidence: `src/duck_orchestrator.py` returns `(error_message, error_code)` but the exception handler only sends `error_code` and never uses `error_message`. Confidence: high. Suggested: either send the full message in debug mode or simplify to return only `error_code` to reduce dead paths.
- Finding: Debug-only S3 listing code is embedded in runtime module `resource_staging.py`, including a hard-coded S3 prefix. Evidence: `src/utils/resource_staging.py` defines `debug_list_s3_prefix` and calls it in a `__main__` block with `s3://stats121-datasets/datasets/`; no other references found. Confidence: medium. Suggested: move to `scripts/` or gate behind an explicit CLI flag to avoid accidental use and keep runtime modules lean.
- Finding: Two different `SqlConfig` TypedDict definitions exist, and the one used by `Config` does not match sqlite-only runtime usage, which can mask config drift. Evidence: `src/utils/config_types.py` defines `SQLConfig` with required `username/password/host/port` fields; `src/storage/sql_connection.py` defines a separate `SqlConfig` and `create_sql_session` branches on `db_type == 'sqlite'` using only `database`. Confidence: high. Suggested: unify the `SqlConfig` type in one place and make sqlite fields optional or a separate union type.
- Finding: Duplicate `SendMessage` protocol definitions can drift and confuse tooling around message signatures. Evidence: `src/utils/protocols.py` defines `class SendMessage(Protocol)`; `src/gen_ai/gen_ai.py` defines a second `class SendMessage(Protocol)` with similar but not identical annotations. Confidence: medium. Suggested: import the shared protocol from `utils.protocols` in `gen_ai` to keep a single source of truth.
- Finding: Two file-sending tools in `TalkTool` appear unreachable from configured tools, making them dead paths. Evidence: `src/armory/talk_tool.py` defines `send_text_file` and `send_json_file`; `rg -n "send_text_file|send_json_file" production-config.yaml prompts src` only finds those definitions and no config tool list references. Confidence: medium. Suggested: remove them if not needed or add to tool lists + docs if they are intended capabilities.
- Finding: Rubric loading logic is duplicated, re-reading and re-parsing the same files in separate methods. Evidence: `src/workflows/assignment_feedback_workflow.py` `_load_rubric` and `_load_rules` each iterate `rubric_files` and call `yaml.safe_load(Path(file).read_text())`. Confidence: medium. Suggested: load rubric files once and share parsed structures to cut IO and reduce drift.
- Finding: Unused config-type declarations suggest dead or planned-but-unwired features. Evidence: `rg -n "MultiAgentSettings|AgentConversationSettings" src` shows only definitions in `src/utils/config_types.py` and no references elsewhere. Confidence: low. Suggested: either wire these types into actual config usage or remove/comment as future-planned to reduce confusion.
